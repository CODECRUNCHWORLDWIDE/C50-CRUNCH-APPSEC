# Exercise 1 — Capstone Threat Model

**Goal:** Produce four real artifacts for your capstone target — a written scope/authorization statement, a data-flow diagram, a full STRIDE table, and a queryable, prioritized risk register in SQLite — that will drive every decision in Exercises 2 and 3.

**Estimated time:** 90 minutes.

```bash
mkdir -p c50-week-12/crunch-ledger && cd c50-week-12/crunch-ledger
# copy config.py, app.py, requirements.txt, schema.sql, seed.py, ci_pipeline.yml from this week's README
python3 -m venv venv && source venv/bin/activate
pip install flask==2.3.2 cryptography

```

If you're hardening your own app instead of Crunch Ledger, confirm it against Lecture 1 Section 1's checklist before continuing — don't start this exercise against a target missing more than one of the five surface-area requirements.

## Task 1 — Write `SCOPE.md`

At the root of your capstone repo, write the scope and authorization statement from Lecture 1 Section 2, filled in for your actual target (not copied verbatim if you're using your own app — the target name, ports, and file list must be real). Every field matters:

```markdown
# Capstone Scope & Authorization

**Target:** <your app name>, running on <host:port>.
**Authorized by:** myself, as the sole owner and operator of this lab environment.
**In scope:** <every file and route this capstone touches>.
**Out of scope:** <anything explicitly excluded, and why>.
**Data:** all data is fictional / synthetic, generated by <your seed script>.
**Duration:** this capstone week only. Re-confirmed before every scan or exploit run.
```

Commit this file before doing anything else — every later exercise, challenge, and the mini-project assumes it exists.

## Task 2 — Draw the data-flow diagram

Using Lecture 1 Section 3's Crunch Ledger diagram as your template (copy it directly if you're using Crunch Ledger; redraw it for your own app's actual routes and data stores if not), produce your own Mermaid `flowchart` covering:

- Every external entity (a human role, an external system).
- Every process (route or major function).
- Every data store.
- At least two trust boundaries — minimum, the network edge and the data-store boundary.

Save it as `THREAT-MODEL.md`'s first section.

## Task 3 — Build the full STRIDE table

For **every process and data flow** in your DFD, ask all six STRIDE questions, even where the answer is "not applicable, because—". Use Lecture 1 Section 4's table as the format:

| STRIDE category | Element | Threat | Maps to (known vuln # or "new gap") |
|---|---|---|---|

Crunch Ledger users: reproduce the lecture's table exactly, then find **at least one additional row** the lecture didn't already give you — reread the app's routes for a threat that isn't one of the eleven numbered vulnerabilities (a Denial-of-Service angle on `/expenses/search`'s unbounded scan is one; find at least one more of your own, in a different STRIDE category, before moving on).

Custom-app users: build this table from scratch against your own DFD — there is no answer key to copy from, which is the actual point of the exercise.

## Task 4 — Build and populate the risk register

```sql
CREATE TABLE risk_register (
    risk_id       INTEGER PRIMARY KEY,
    stride_category TEXT NOT NULL CHECK (stride_category IN
        ('spoofing','tampering','repudiation','info_disclosure','dos','elevation')),
    element       TEXT NOT NULL,
    description   TEXT NOT NULL,
    source        TEXT NOT NULL CHECK (source IN ('threat_model','scan','manual_review')),
    likelihood    TEXT NOT NULL CHECK (likelihood IN ('low','medium','high')),
    impact        TEXT NOT NULL CHECK (impact IN ('low','medium','high')),
    priority      TEXT,
    status        TEXT NOT NULL DEFAULT 'open' CHECK (status IN ('open','mitigated','accepted')),
    notes         TEXT
);
```

Insert one row per STRIDE table entry from Task 3 — every row, including the ones that map to a known numbered vulnerability, because the register is what drives Exercise 2's build order, not the README's vulnerability list. Then compute priority:

```sql
UPDATE risk_register
SET priority = CASE
    WHEN likelihood = 'high' AND impact = 'high' THEN 'P0'
    WHEN likelihood = 'high' OR  impact = 'high' THEN 'P1'
    WHEN likelihood = 'medium' AND impact = 'medium' THEN 'P2'
    ELSE 'P3'
END;
```

Verify with a query, not a glance at the INSERT statements:

```bash
sqlite3 risk_register.db "SELECT priority, COUNT(*) FROM risk_register GROUP BY priority ORDER BY priority;"
sqlite3 risk_register.db "SELECT priority, element, description FROM risk_register ORDER BY priority, risk_id;"
```

## Done when…

- [ ] `SCOPE.md` exists, is accurate to your actual target, and has all six fields filled in with real values (not placeholders).
- [ ] Your DFD renders correctly as a Mermaid diagram and shows at least two trust boundaries.
- [ ] Your STRIDE table has at least one row per STRIDE category and includes at least one finding **not** already named in the README's vulnerability table (Crunch Ledger users) or built entirely from your own analysis (custom-app users).
- [ ] `risk_register` is populated, every row has a computed `priority`, and `SELECT priority, COUNT(*) FROM risk_register GROUP BY priority` returns a sensible distribution (not everything `P0` — that's a sign your likelihood/impact ratings aren't actually differentiating anything).
- [ ] At least one row has `status = 'accepted'`, with a `notes` field that could survive being read aloud to a stranger as a justification.

## Stretch

- Cross-reference every STRIDE row against the OWASP Top 10 category it most resembles (Week 3) — this cross-reference becomes useful evidence in the mini-project's final report.
