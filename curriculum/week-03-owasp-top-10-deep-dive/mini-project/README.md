# Mini-Project — A Full Top 10 Evidence Report for `crunch-notes`

> Close out every remaining category. By Saturday evening, all ten OWASP Top 10 flaws deliberately built into `crunch-notes` are demonstrated, fixed, re-tested, and recorded — one findings database, ten rows, ten proofs.

**Estimated time:** 4–4.5 hours, best done Saturday after the exercises and challenges.

This is the week's capstone, and it's built to be genuinely completable: `crunch-notes` contains **exactly** one representative flaw per category (the table in the week README), and by this point you've already handled several of them:

- **Exercises 1–2** demonstrated, fixed, and re-tested A01 (×2) and A05.
- **Challenge 2** demonstrated, fixed, and re-tested three more of your choosing from {A02, A03, A04, A06, A07, A08}.
- That leaves **at most three** categories you haven't personally fixed yet — this project's job is to close every one of those out, so that by the end, all ten rows in your `findings.db` show `status = 'remediated'` with a `retest_result = 'pass'`.

If you chose a different three in Challenge 2 than another student, your "remaining three" will differ — that's fine. Everyone ends this project in the same place: **all ten done.**

---

## Deliverable

A directory in your portfolio `c50-week-03/mini-project/` containing:

1. Your final `crunch-notes/app.py` — all ten flaws fixed.
2. Your final `crunch-notes/schema.sql` — including the A04 `CHECK` constraint.
3. Your final `crunch-notes/requirements.txt` — upgraded, `pip-audit`-clean versions.
4. `findings.db` — ten rows, one per OWASP category, every one `status = 'remediated'`.
5. `evidence-report.md` — a human-readable report generated *from* `findings.db`, one section per category (template below).
6. `notes.md` — a short reflection (see the end).

## Step 1 — Inventory what's already closed

Query your own `findings.db` before doing anything else:

```sql
SELECT owasp_category, title, status, retest_result FROM findings ORDER BY owasp_category;
```

List, in `evidence-report.md`, which categories are already `remediated` (from Exercises 1–2 and Challenge 2) and which are still open. That second list is your task list for this project.

## Step 2 — Close out every remaining category

For each category still open, work the full cycle exactly as you have all week:

1. **Demonstrate** it against your current, unmodified `crunch-notes` — capture the request/response.
2. **Remediate** it at the source — in `app.py`, `schema.sql`, or `requirements.txt`, whichever the flaw actually lives in.
3. **Re-test** — confirm the attack now fails *and* the legitimate use case still works.
4. **Record** it — insert or update the row in `findings.db` with real timestamps, `remediation_summary`, `retested_at`, `retest_result`, and `status = 'remediated'`.

You've already done this exact cycle for at least five categories this week. There is no new mechanic here — this step is about **completeness and rigor**, not new material.

## Step 3 — Verify the whole app end to end

Once every category shows `remediated` in the database, prove it against the *running app*, not just the database rows — a database can claim something is fixed while the code says otherwise if you were sloppy about which file you edited.

```bash
# restart clean
pkill -f "python3 app.py" 2>/dev/null
python3 seed.py   # fresh db
python3 app.py &
```

Re-run **every** attack from Lectures 1–3 against this freshly restarted app, one after another, and confirm every single one now fails the way its fix predicts (a 404, a 403, an empty result set, a 400, a rejected insert, or a refused deserialization — whichever your fix produces). Save the full transcript as `full-retest-transcript.md`.

## Step 4 — Generate the evidence report

`evidence-report.md` — one section per category, generated by reading your own `findings.db` (write a small Python script if you want to automate the formatting; doing it by hand is fine too, as long as the numbers come from the database, not from memory):

```markdown
## A01:2021-Broken Access Control

**Finding:** IDOR on GET /notes/<note_id>
**Risk score:** 12 (likelihood 4 × impact 3)
**Demonstrated:** 2026-07-20T14:02:00Z — alice's session read bob's note (id 2)
**Remediated:** 2026-07-20T14:40:00Z — added `AND user_id = ?` to the ownership query
**Re-tested:** 2026-07-20T14:45:00Z — PASS (attack now returns 404; owner access unaffected)

... (repeat for all ten)
```

## Rules

- **All ten categories, no exceptions.** A report with nine is not this project — every category `crunch-notes` was built to teach must show `remediated`.
- **Every remediation must be re-tested in both directions** (attack fails, legitimate use still works) — a `retest_result = 'pass'` you didn't actually verify is worse than an honest `'fail'` you're still working on.
- **Evidence, not assertions.** Every "demonstrated" claim needs an actual request/response behind it, referenced in `evidence-report.md` even if the full text lives in a linked file.
- **No spreadsheet, ever.** `findings.db` is the single source of truth for this report; `evidence-report.md` is generated *from* it, not maintained separately by hand.

## Reflection (`notes.md`, ~250 words)

1. Which of the ten categories took the longest to fix correctly, and what made it hard — the concept, or getting the code exactly right?
2. Pick one fix and explain what would have happened if you'd only patched the *symptom* you first noticed instead of the *root cause* (e.g., blocking one specific injection payload instead of parameterizing the query).
3. Now that `findings.db` holds all ten, write the one SQL query you'd run every morning if this were a real production app you were responsible for, and why that's the one.
4. Which category do you feel least confident you could reproduce, unassisted, on a completely different app next week? That's your signal for what to re-read before Week 4.

You just did it against code you understand completely, which is the right way to build the habit before Week 11 has you do a version of this against code you've never seen before.
