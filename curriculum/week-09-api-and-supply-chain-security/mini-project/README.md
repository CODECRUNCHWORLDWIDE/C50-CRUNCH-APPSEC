# Mini-Project — Secure a Vulnerable Lab API and Its Supply Chain

> Close every remaining gap, on both halves of the week. By Saturday evening, `crunch-tasks-api` enforces object- and function-level authorization, blocks mass assignment and excessive data exposure, and rate-limits its sensitive endpoints — and its dependency tree is pinned, hash-verified, SBOM'd, scanned, and every finding is recorded in a database.

**Estimated time:** 4 hours, best done Saturday after the exercises and challenges.

This is the week's capstone, and by this point you've already done almost all of the actual work:

- **Exercises 1–2** demonstrated, fixed, and re-tested the BOLA and both mass-assignment flaws.
- **Exercise 3** generated and scanned an SBOM, and built your first `dependency_findings` rows.
- **Challenge 1** closed BFLA, excessive data exposure, and missing rate limiting.
- **Challenge 2** proved hash-pinning (or index-priority) stops a dependency-confusion or typosquat attack.

This project's job is to **verify the whole thing holds together as one coherent system**, produce a single findings database covering both the API and the supply chain, and write the report a real security team would actually hand to an engineering lead.

---

## Deliverable

A directory in your portfolio `c50-week-09/mini-project/` containing:

1. Your final `crunch-tasks-api/app.py` — every API flaw fixed, using the reusable helpers/serializers/limiter from Challenge 1, not narrow patches.
2. Your final `crunch-tasks-api/requirements.txt` and `requirements.lock.txt` (hash-pinned, per Lecture 3 Section 4).
3. `sbom.json` — regenerated from the final, hash-pinned dependency set.
4. `security-findings.db` — one SQLite database with **two** tables: `api_findings` (five rows — one per API Top 10 category from the week README's table) and `dependency_findings` (every scan result from Exercise 3 and Challenge 2), every row's `status` reflecting its true, currently-verified state.
5. `evidence-report.md` — a human-readable report generated *from* `security-findings.db`, one section per finding.
6. `notes.md` — a short reflection (see the end).

## Step 1 — Inventory what's already closed

Query both tables in `security-findings.db` before doing anything else:

```sql
SELECT category, endpoint, status FROM api_findings ORDER BY category;
SELECT package_name, advisory_id, status FROM dependency_findings ORDER BY package_name;
```

List, in `evidence-report.md`, which findings are already `remediated`/`upgraded` and which are still open. That list is this project's task list — for most students, this should be short or empty, since Challenge 1 and Challenge 2 already did the closing work. If anything is still open, close it now using the exact demonstrate → remediate → re-test cycle you've used all week.

## Step 2 — Full clean-restart verification of the API

A findings database can claim something is fixed while the running code says otherwise, if you edited the wrong file or forgot to restart. Prove it against the live process:

```bash
pkill -f "python3 app.py" 2>/dev/null
python3 seed.py    # fresh db
python3 app.py &
```

Re-run **every** attack from Lectures 1–2, Exercises 1–2, and Challenge 1 against this freshly restarted app, one after another: the BOLA read, both mass-assignment attempts, the BFLA delete, the excessive-exposure check, and the rate-limit loop. Save the full transcript as `full-retest-transcript.md`. Every single attack must now fail exactly the way its fix predicts (a 404, a 403, a 400, or a 429) — and every legitimate action (Alice reading her own task, Bob deleting a task as admin, a normal login) must still succeed.

## Step 3 — Full verification of the supply chain

```bash
pip install --require-hashes -r requirements.lock.txt
pip-audit -r requirements.txt --format=cyclonedx-json -o sbom.json
pip-audit -r requirements.txt
```

Confirm the regenerated `sbom.json` matches your currently-installed, hash-pinned dependency set, and that the `pip-audit` scan output matches what's recorded in `dependency_findings` — if a package version changed since Exercise 3 (say, you upgraded Flask during Challenge 1), update the corresponding row's `status` to `'upgraded'` rather than leaving a stale `'open'` row for a version you no longer run.

## Step 4 — Generate the evidence report

`evidence-report.md`, one section per finding, generated by reading your own database (a small Python formatting script is fine; doing it by hand from real query output is fine too — the numbers must come from the database, not memory):

```markdown
## API1:2023 Broken Object-Level Authorization

**Finding:** BOLA on GET /api/v1/tasks/<id>
**Risk score:** 12 (likelihood 4 x impact 3)
**Demonstrated:** 2026-07-20T09:12:00Z — alice's token read bob's task (id 3)
**Remediated:** 2026-07-20T09:40:00Z — added AND user_id = ? to the task lookup
**Re-tested:** 2026-07-20T09:45:00Z — PASS (attack now returns 404; owner access unaffected)

... (repeat for all five API findings and every dependency finding)
```

## Rules

- **Both tables, all rows, no exceptions.** A report covering only the API half or only the supply-chain half is not this project.
- **Every remediation re-tested in both directions** — attack fails, legitimate use still works — a `status` you didn't actually verify against the running app is worse than an honestly-still-open row.
- **Evidence, not assertions.** Every "demonstrated" and "re-tested" claim needs a real request/response or terminal transcript behind it.
- **No spreadsheet, ever.** `security-findings.db` is the single source of truth; `evidence-report.md` is generated *from* it.

## Reflection (`notes.md`, ~250 words)

1. Which felt more natural to fix: the API authorization flaws (same discipline as Weeks 3 and 6, new transport) or the supply-chain issues (a genuinely new kind of "who do I trust")? Why?
2. Pick one API fix and explain what would have happened if you'd only patched the *specific attack you tried* instead of the *general pattern* (e.g., blocking one specific field name instead of using an allowlist).
3. Now that `security-findings.db` holds both API and dependency findings, write the one query you'd run every Monday morning if this were a real production service you were responsible for, and explain why that's the one.
4. Which of this week's five API categories, or which supply-chain concept, do you feel least confident you could recognize unassisted on a completely different API next week? That's your signal for what to re-read before Week 10.
